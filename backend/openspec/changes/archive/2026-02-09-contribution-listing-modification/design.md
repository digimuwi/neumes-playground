## Context

The backend stores contributions as `contributions/<uuid>/` directories, each containing an image file and `annotations.json`. The existing `storage.py` has `save_contribution()` and `list_contributions()`. The existing `list_contributions()` returns `(id, path)` tuples but doesn't read annotation content. There are no endpoints to retrieve or update individual contributions.

The frontend needs to browse contributions, load one into the editor, modify annotations, and save changes back — all via REST API.

## Goals / Non-Goals

**Goals:**
- Expose contribution listing, retrieval, and update via REST endpoints
- Return base64-encoded images so the frontend can display them without a separate image endpoint
- Preserve image metadata when updating annotations (only annotations change, never the image)
- Validate contribution IDs to prevent path traversal attacks

**Non-Goals:**
- Deleting contributions (out of scope for this change)
- Uploading a new image for an existing contribution (use POST /contribute for that)
- Pagination or filtering of the contributions list (not needed at current scale)
- Serving images via a dedicated streaming endpoint (base64 in JSON is sufficient)

## Decisions

### 1. Base64 image encoding in GET response

**Decision**: Return the image as a data URL (`data:image/jpeg;base64,...`) embedded in the JSON response.

**Alternatives considered**:
- Separate `GET /contributions/{id}/image` endpoint — adds complexity, requires two requests from frontend
- Serve static files from contributions directory — security risk, CORS complexity

**Rationale**: Contribution images are manuscript crops, typically a few hundred KB to a few MB. Base64 encoding adds ~33% overhead but keeps the API simple — one request returns everything the frontend needs to render the editor.

### 2. ContributionDetail as a Pydantic response model

**Decision**: Define a `ContributionDetail` Pydantic model for `GET /contributions/{id}` rather than returning a raw dict.

**Rationale**: Every other endpoint uses typed Pydantic models. Consistency aids documentation (auto-generated OpenAPI schema) and validation.

### 3. UUID validation for path safety

**Decision**: Validate that `{id}` path parameters match UUID format before constructing file paths.

**Rationale**: Contribution IDs are always UUIDs (generated by `uuid.uuid4()`). Rejecting non-UUID strings prevents path traversal (e.g., `../../etc/passwd`) without needing generic path sanitization.

### 4. PUT replaces annotations only

**Decision**: `PUT /contributions/{id}` accepts a `ContributionAnnotations` JSON body and replaces `lines` and `neumes` in `annotations.json`, preserving the `image` metadata block.

**Alternatives considered**:
- Accept full `annotations.json` including image block — risk of frontend sending stale/wrong image metadata
- PATCH with partial updates — unnecessary complexity, the frontend always has the full annotation state

**Rationale**: The image never changes during an update. Reading the existing `image` block from disk and merging it with the new annotations is safe and prevents accidental metadata corruption.

### 5. Reuse ContributionAnnotations for PUT body

**Decision**: The PUT request body uses the same `ContributionAnnotations` model as POST `/contribute`.

**Rationale**: The annotation format is identical — `lines` with `syllables` and `neumes` with `bbox`. Reusing the model avoids duplication and ensures the same validation rules apply.

## Risks / Trade-offs

- **[Large images slow down GET response]** → Base64 encoding large images increases response size and latency. Acceptable at current scale (manuscript crops, not full pages). If this becomes a problem, add a separate image endpoint later.
- **[No concurrency control on PUT]** → Two simultaneous updates could overwrite each other. Acceptable for a single-user tool. If needed later, add ETags or last-modified checks.
- **[List endpoint reads all annotations.json files]** → O(n) file reads for n contributions. Fine for dozens of contributions. If it grows to hundreds, add caching or summary metadata.
