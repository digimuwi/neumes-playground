# Neumes Playground MEI Profile

This is the MEI dialect used to store contributions on disk and transmit them
over the wire. Both the backend (`htr_service.contribution.mei_io`) and the
frontend (`src/utils/meiIO.ts`) implement reader/writer pairs against this
spec. The fixtures in `spec/fixtures/` are the contract — both halves must
round-trip them losslessly.

## Format

- **MEI version**: 5.0
- **Namespace**: `http://www.music-encoding.org/ns/mei`
- **File extension on disk**: `annotations.mei` (sibling of `image.{jpg,png}`)
- **Wire encoding**: UTF-8 XML, with XML declaration. Multipart form field
  named `mei` for `POST /contribute`; JSON-wrapped string `{ "mei": "..." }`
  for `PUT /contributions/{id}`.

## Document structure

```xml
<?xml version='1.0' encoding='UTF-8'?>
<mei xmlns="http://www.music-encoding.org/ns/mei" meiversion="5.0">
  <meiHead>
    <fileDesc>
      <titleStmt><title>Neumes Playground contribution</title></titleStmt>
      <pubStmt/>
    </fileDesc>
  </meiHead>
  <music>
    <facsimile>
      <surface xml:id="surface-1" ulx="0" uly="0"
               lrx="{image_width}" lry="{image_height}">
        <graphic target="{image_filename}"
                 width="{image_width}px" height="{image_height}px"/>
        <!-- one <zone> per syllable, neume, and line boundary -->
      </surface>
    </facsimile>
    <body>
      <mdiv>
        <score>
          <section>
            <staff>
              <layer>
                <!-- <lb> + <syllable> children, in reading order -->
              </layer>
            </staff>
          </section>
        </score>
      </mdiv>
    </body>
  </music>
</mei>
```

## Polygon encoding

All polygonal regions (syllable boundaries, line boundaries, neume bboxes)
are stored as `<zone>` elements with **both**:

- `@points="x1,y1 x2,y2 ..."` — full polygon (lossless)
- `@ulx`, `@uly`, `@lrx`, `@lry` — bounding rectangle of the polygon (for
  legacy MEI consumers that don't read `@points`)

```xml
<zone xml:id="zone-syl-{uuid}"
      ulx="100" uly="200" lrx="180" lry="240"
      points="100,200 180,200 175,240 105,240"/>
```

Coordinates are integer pixels in the image's coordinate system (origin
top-left).

Neume zones currently store only an axis-aligned rect (the OCR backend
emits bboxes), so for neumes `@points` describes the four corners of the
rect. The format is uniform regardless.

## xml:id conventions

- Surface: `surface-1`
- Syllable zone: `zone-syl-{uuid}`
- Neume zone: `zone-n-{uuid}`
- Line zone: `zone-line-{N}` (1-indexed)
- Syllable element: `syl-{uuid}`
- Neume element: `n-{uuid}`

The `{uuid}` parts embed the runtime UUID generated by either side. **Readers
must regenerate fresh UUIDs on load** — round-trip identity of UUIDs is not
guaranteed and must not be relied on. The xml:ids exist only to make the
`@facs` references inside one document resolve.

## Layer content (reading order)

The `<layer>` contains a flat sequence of `<lb>` and `<syllable>` elements
in reading order:

```xml
<layer>
  <lb n="1" facs="#zone-line-1"/>
  <syllable xml:id="syl-{uuid}" facs="#zone-syl-{uuid}">
    <syl wordpos="i">Do-</syl>
    <neume xml:id="n-{uuid}" type="punctum" facs="#zone-n-{uuid}"/>
  </syllable>
  <syllable .../>
  <lb n="2" facs="#zone-line-2"/>
  ...
</layer>
```

## `<lb>` (line beginning)

- `@n` — 1-indexed line number, monotonically increasing
- `@facs` — points to a `<zone>` containing the line boundary polygon

A line is **the run of `<syllable>`s between this `<lb>` and the next** (or
end-of-layer). On load, the `<lb>` zone restores `LineBoundary.boundary`
and the syllable IDs in that run populate `LineBoundary.syllableIds`.

## `<syllable>` and `<syl>`

```xml
<syllable xml:id="syl-{uuid}" facs="#zone-syl-{uuid}">
  <syl wordpos="i|m|t|s">{text}</syl>
  <neume .../>
  <neume .../>
</syllable>
```

- `@facs` references the zone holding the syllable's polygon boundary.
- `<syl>` text **never** contains the trailing `-` hyphen used internally
  by the editor. The hyphen is reconstructed on load from `@wordpos`:
  - `i` (initial) → text gets `-` appended
  - `m` (middle)  → text gets `-` appended
  - `t` (terminal) → no hyphen
  - `s` (single) → no hyphen
- Empty syllables are written as `<syl/>` (no text, no `@wordpos`).
- `@wordpos` is omitted when text is empty.

### Empty syllable carriers (orphan neume convention)

Neumes that, by spatial assignment, have no owning syllable on a given
line are nested under a synthetic empty syllable on that line:

```xml
<syllable xml:id="syl-orphan-{lineN}" facs="#zone-line-{lineN}">
  <syl/>
  <neume .../> <!-- the orphan -->
</syllable>
```

The synthetic syllable's `@facs` reuses the line's zone (the orphan has
no syllable boundary of its own). On load, neumes inside any syllable
whose `<syl>` is empty AND whose `@facs` points at a `zone-line-*` are
restored as flat neume `Annotation`s with no syllable association — they
are just neumes the geometry-based assignment couldn't place.

### Contributions with no syllables at all

A contribution may have neumes but zero text lines (the user labeled
neumes only). In that case the writer emits exactly one synthetic line
covering the full image bounds plus one synthetic empty syllable
carrying every neume:

```xml
<surface ...>
  <zone xml:id="zone-line-1"
        ulx="0" uly="0" lrx="{W}" lry="{H}"
        points="0,0 {W},0 {W},{H} 0,{H}"/>
  <!-- neume zones -->
</surface>
...
<layer>
  <lb n="1" facs="#zone-line-1"/>
  <syllable xml:id="syl-orphan-1" facs="#zone-line-1">
    <syl/>
    <neume .../>
    <!-- ... all neumes ... -->
  </syllable>
</layer>
```

On load, this restores to `lineBoundaries: []` and a flat list of neume
annotations. The structural test for "this is a synthetic line, drop it"
is: the line's run contains exactly one syllable, that syllable's `<syl>`
is empty, and its `@facs` points at the same `zone-line-*` as the `<lb>`
itself (i.e., no real syllable boundary).

The same test applies to per-line orphan carriers in lines that DO have
real syllables: their parent line is real (kept), but the orphan
carrier itself is dropped on load — its neume children are reattached
flat.

## `<neume>`

```xml
<neume xml:id="n-{uuid}" type="{neume_type}" facs="#zone-n-{uuid}"/>
```

- `@type` is the canonical neume type string (e.g., `punctum`, `clivis`).
  Any string is accepted — validation against the neume registry happens
  elsewhere.
- `@facs` references the neume's bounding-rect zone.

## Canonical serialization

For deterministic ETag computation (sha256 of file bytes), serializers MUST:

- Use UTF-8 with XML declaration `<?xml version='1.0' encoding='UTF-8'?>`
- Emit attributes in **sorted order** within each element
- Use 2-space indentation, LF (`\n`) line endings
- Close elements with self-closing syntax when empty
- Not emit XML comments
- Not emit `xml:lang`, `xmlns:xml`, or other meta attributes unless required

The backend computes the version hash by re-serializing the parsed MEI
through its canonical writer before hashing. **The frontend's MEI is never
hashed directly** — the backend always re-canonicalizes first. This avoids
serialization-drift differences between Python's `xml.etree` and the
browser's `XMLSerializer`.

## Round-trip semantics

1. **Annotation IDs**: regenerated on load. Round-tripping does not
   preserve UUIDs.
2. **Neume↔syllable assignments**: the structural nesting in the layer
   IS the assignment. On load, neumes are reattached as flat
   `Annotation` instances; the assignment is recovered on demand by
   `computeNeumeAssignments()` (frontend) or by walking the parsed MEI
   (backend, e.g., for training export). For training data this means
   the assignment is committed at save time.
3. **Empty syllables** preserved as `<syl/>` with no `@wordpos`; their
   text is restored as the empty string.
4. **Hyphen reconstruction** from `@wordpos` is mandatory on load.
5. **Reading order** within a line is left-to-right by syllable polygon
   center-X. Writers MUST emit syllables in reading order; readers MUST
   NOT rely on any other ordering.
6. **Line ordering** is top-to-bottom by line zone center-Y; `@n`
   reflects this ordering.

## What is NOT preserved

- Confidence scores from the OCR backend (only stored at recognition
  time, not contribution time)
- Baselines (lines have boundary polygons but no separate baseline)
- Document-level metadata (title, source, etc.) beyond the boilerplate
  `<meiHead>`
